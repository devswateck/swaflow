# Epic 8 Retrospective

Date: 2026-07-17

## Epic Review

### Outcome

- Epic 8 completed all 7 stories.
- Epic status in sprint tracking is `done`.
- No previous epic retrospective artifact was found, so this is the first retro in this chain.

### What Went Well

- Review-driven hardening worked: each story fed the next one with a narrower, more precise contract.
- The team consistently converted review findings into deterministic regressions instead of one-off fixes.
- Tenant isolation and permission enforcement became clearer as the epic progressed.
- The hardest bugs were handled by tightening contracts at the smallest correct boundary, not by broad refactors.
- Backend and frontend contracts were kept aligned when the UI depended on backend flags or snapshots.

### What Was Hard

- Several stories needed multiple review-fix loops before the contract was stable.
- The same pattern surfaced more than once: a UI label or fallback implied behavior that backend no longer supported.
- Permission gaps were easier to miss on read routes than on write routes.
- Stale-state problems kept reappearing in different shapes: inbox selection, appointment snapshots, and webhook fallback logic.
- The shell environment still did not have `fastapi`, which blocked `pytest` verification in this workspace.

### Key Lessons

Amelia (Developer): "The strongest pattern in Epic 8 was to fix the contract first, then lock it with a regression."

Alice (Product Owner): "And when a public-facing label or fallback changes, the UI copy has to change with it. Otherwise we lie to the operator."

Charlie (Senior Dev): "The smallest reliable fix was usually a route-level guard or a deterministic snapshot key, not a bigger abstraction."

Dana (QA Engineer): "The best regressions were the ones that proved the bad behavior could not come back, not just that the happy path worked."

- Contract changes must update backend, UI, and tests together.
- Read routes need the same permission scrutiny as write routes.
- Snapshot/version fields should be deterministic and explicit when multiple events can collide.
- Regression stories work best when they preserve the original bug shape, then strengthen it until the fix cannot be bypassed.

### Evidence-Based Patterns

- Inbox work exposed stale-state races and selection drift.
- Assignment and permission work exposed the need for route-level authorization, not only helper-level checks.
- WhatsApp and integrations work repeatedly exposed secret handling and safe-public-contract issues.
- Appointment hydration work showed that snapshot selection needs a stable version marker, not only a timestamp.
- The final regression story confirmed that review findings themselves can become test fixtures for future hardening.

### Summary

Epic 8 succeeded as a hardening epic. The team did not add broad product surface; it tightened fragile paths, reduced contract drift, and left behind a stronger regression net.

## Next Epic Preparation

### Current State

- No Epic 9 is defined in planning artifacts yet.
- The most useful preparation is to keep the regression-first pattern for any future epic that touches public contracts, permissions, or snapshots.

### Preparation Recommendations

1. Keep a contract checklist for every story that changes UI copy, safe flags, or public API fallbacks.
2. Require explicit read-route permission checks whenever a module gate changes.
3. Continue using deterministic snapshot/version markers for any feature that can be rehydrated from events.
4. Keep backend syntax/build validation in the completion gate when a story touches both backend and frontend.
5. Preserve the pattern of writing the smallest regression that reproduces the bug shape before widening coverage.
