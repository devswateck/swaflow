# Decision Log: PRD Swaflow

Este archivo registra decisiones, cambios y overrides del PRD.

## Decisions

- 2026-06-08: Se inicio un PRD nuevo para Swaflow.
- 2026-06-08: Se cargara `_bmad-output/project-context.md` como contexto persistente de producto, stack y restricciones tecnicas vigentes.
- 2026-06-08: El usuario confirmo que el PRD debe usar `proyecto_saas_ia_comercial_multitenant.md` como insumo de producto.
- 2026-06-08: En conflictos tecnicos entre el documento historico y `project-context.md`, prevalece `project-context.md`: MySQL via `mysql+pymysql`, no PostgreSQL, y Redis sin colas criticas activas.
- 2026-06-08: El alcance de producto a documentar es una plataforma web SaaS multi-tenant para gestionar chats de Meta/WhatsApp, IA comercial, ventas, inventario, ordenes, citas, funnels, integraciones y administracion de superusuario.
- 2026-06-08: El modulo de IA del backend fue validado como base existente: agente unico por tenant, prompt del sistema, objetivo, guia conversacional, reglas de seguridad, FAQs, plantillas interactivas, contexto de catalogo/inventario y runtime con respuesta JSON.
- 2026-06-08: Se investigo Embedded Signup/Facebook Login for Business como alternativa para simplificar la conexion de WhatsApp.
- 2026-06-08: El usuario eligio `Coaching path` para construir el PRD seccion por seccion.
- 2026-06-08: El usuario confirmo que el PRD es para lanzamiento comercial SaaS, no solo operacion interna.
- 2026-06-08: Para la primera version comercial, SuperUsuario queda fuera del alcance V1 y se difiere a V2. Los demas modulos mencionados entran en V1 porque ya estan adelantados.
- 2026-06-08: Se adopto Vision + Features como punto de entrada del coaching por tratarse de plataforma SaaS B2B modular.
- 2026-06-09: Cada tenant tendra un usuario admin principal. Los usuarios adicionales tendran costo adicional de gestion y no accederan por defecto a WhatsApp, IA, Funnels, Integraciones ni Configuracion.
- 2026-06-09: El admin del tenant podra habilitar acceso por usuario a modulos restringidos mediante checks en Configuracion, sin cambiar el rol del usuario.
- 2026-06-09: El superadmin pertenece a Swateck y tendra acceso ilimitado a todos los tenants. Esto es distinto del panel/aplicacion SuperUsuario, que sigue diferido a V2.
- 2026-06-09: Los usuarios adicionales tendran acceso por defecto solo a Dashboard, Inbox, Productos, Inventario, Ordenes y Citas.
- 2026-06-09: En Configuracion, el admin debe ver un mensaje resaltado indicando que cada usuario adicional tendra costo mensual.
- 2026-06-09: El criterio principal de exito de V1 comercial es que todo el flujo funcione de punta a punta, no solo modulos aislados.
- 2026-06-09: El Dashboard V1 debe permitir filtrar metricas por rango de fechas, asesor/usuario, estado de chat, funnel y producto.
- 2026-06-09: La desactivacion de IA debe ser por chat/conversacion, reversible, y al reactivarse la IA debe retomar con contexto de lo conversado por el humano.
- 2026-06-09: Regla de asignacion de chats V1: si solo existe admin, el admin puede gestionar todos los chats; si existe un usuario adicional, todos los chats se asignan a ese usuario; si hay mas de un usuario adicional, cada usuario puede tomar chats y los chats tomados dejan de estar disponibles para otros usuarios.
- 2026-06-09: El admin del tenant podra asignar chats manualmente a un usuario.
- 2026-06-09: El admin del tenant podra reasignar un chat aunque ya haya sido tomado o este asignado a otro usuario.
- 2026-06-09: WhatsApp V1 mantiene la configuracion tecnica actual de WhatsApp Cloud API; WhatsApp V2 debe simplificar conexion con autenticacion/popup de Meta.
- 2026-06-09: En V1, todos los productos deben existir en Meta. Swaflow no debe crear productos adicionales ni ser fuente primaria del catalogo.
- 2026-06-09: La disponibilidad base del inventario debe venir de Meta. Swaflow debe reflejarla y usar reservas operativas propias para ordenes en proceso cuando aplique.
- 2026-06-09: Las ordenes deben mostrar estados en espanol para cliente/admin, aunque internamente puedan mantener codigos estables en ingles.
- 2026-06-09: Las citas deben poder crearse por IA o manualmente desde Inbox. Si no hay calendario integrado, la cita igual debe existir y verse en el modulo Citas.
- 2026-06-09: Siempre debe existir un funnel inicial de bienvenida para mensaje inicial y captura de datos del cliente. Luego la IA puede asignar cualquier funnel segun intencion y criterios configurados por el cliente.
- 2026-06-09: El cliente/admin puede crear funnels y definir caracteristicas/criterios de asignacion.
- 2026-06-09: Configuracion IA V1 debe incluir horario de atencion, reglas de autonomia, politicas comerciales verificadas, reglas de escalamiento, categorias/productos prioritarios o restringidos, modo de prueba, estado de publicacion y objetivo comercial.
- 2026-06-09: El horario de atencion de IA debe permitir configurar un horario para lunes a viernes y otro diferente para sabado y domingo.
- 2026-06-09: Integraciones nativas V1 se limitan a pasarela de pagos, calendario, correo/notificaciones y webhooks salientes. Integraciones perifericas quedan para n8n via webhooks.
- 2026-06-09: Configuracion V1 debe incluir datos de empresa, logo/marca, zona horaria, moneda, usuarios/permisos, seguridad/cambio de contrasena y carga de foto de banner/profile para correos a clientes.
- 2026-06-09: Roles/permisos se formalizaron como FRs implementables, no solo como narrativa de producto.
- 2026-06-09: Los NFRs transversales deben incluir tiempos de respuesta como expectativa explicita de producto.
- 2026-06-09: Los tiempos de respuesta V1 se definiran como linea base recomendada y ajustable, no como SLA contractual final.
- 2026-06-09: Se completaron NFRs transversales de seguridad, multi-tenant, auditoria, privacidad, disponibilidad y confiabilidad de integraciones.
- 2026-06-09: Se inicio captura de journeys V1, comenzando por compra completa por WhatsApp con IA, backend, orden y pago.
- 2026-06-09: Swaflow debe servir a negocios que venden productos, negocios que solo agendan citas y negocios mixtos.
- 2026-06-09: El flujo inicial debe capturar datos principales del cliente y permitir elegir opciones como compra de producto, consulta de productos o agendamiento, mediante menus/listas configurables.
- 2026-06-09: Si expira un link de pago, la IA puede hacer seguimiento para preguntar si el cliente desea continuar con el pago o agregar otro producto.
- 2026-06-09: Se agregaron FRs explicitos para modo de negocio del tenant y seguimiento de pagos expirados, sin renumerar IDs existentes.
- 2026-06-09: Para agendamiento, cuando el cliente solicite cita la IA debe preguntar preferencia manana/tarde, validar disponibilidad y proponer 3 horarios posibles en dias diferentes.
- 2026-06-09: Si hay calendario integrado, la disponibilidad de citas se valida contra ese calendario; si no, se valida contra citas internas y horario establecido por el comercio en el modulo Citas.
- 2026-06-09: El horario operativo debe ser unico y compartido: el mismo horario aplica para IA y para disponibilidad de citas.
- 2026-06-09: Se aprobo capturar journey de configuracion inicial del admin cubriendo empresa, modo de negocio, horario, usuarios/permisos, WhatsApp, IA, funnels, integraciones, catalogo Meta e inventario.
- 2026-06-09: Retencion V1: mensajes, eventos, archivos y gestiones se conservan indefinidamente mientras el tenant este activo.
- 2026-06-09: Cuando un cliente/tenant se retire, Swaflow debe entregar un paquete con todas las gestiones realizadas en la plataforma.
- 2026-06-09: Links de pago expiran por defecto a los 120 minutos, pero el tiempo de expiracion debe ser configurable en la integracion de pagos.
- 2026-06-09: Pasarela V1 no debe quedar atada a Wompi; el cliente debe poder implementar/configurar la pasarela que desee dentro del modelo soportado por Swaflow.
- 2026-06-09: Calendario V1 debe contemplar Google Calendar o Microsoft Calendar.
- 2026-06-09: Funnel de bienvenida debe capturar Nombre, Correo y Ciudad. Telefono se obtiene del chat e intencion la completa el funnel.
- 2026-06-09: Auditoria preliminar completo Decision Audit e Input Reconciliation; se detecto como pregunta bloqueante la regla base de disponibilidad de citas.
- 2026-06-09: Disponibilidad de citas V1: duracion por defecto 1 hora configurable en Citas; manana 08:00-12:00; tarde 14:00-18:00; anticipacion minima desde manana; horizonte maximo proximos 7 dias.
- 2026-06-09: Provisionamiento V1: cada tenant y su admin principal son creados por Swateck; self-service signup de tenants queda fuera de V1.
- 2026-06-09: Pasarelas V1 se soportan mediante adaptadores/proveedores configurados o certificados por Swateck, con contrato comun para link de pago, expiracion, webhook, estados e idempotencia.
- 2026-06-09: Autoasignacion con exactamente un usuario adicional se mantiene por defecto, pero el admin puede desactivarla para que los chats queden disponibles/no asignados.
- 2026-06-09: Exportacion al retiro del tenant sera un ZIP con un TXT por modulo, incluyendo interacciones/gestiones delimitadas por pipe `|`.
- 2026-06-09: Se genero contrato tecnico detallado para adaptadores de pasarela en `addendum.md`.
- 2026-06-09: Cadencia de seguimiento significa frecuencia/limite de mensajes de IA despues de un link expirado; queda pendiente aprobar la regla recomendada.
- 2026-06-09: Seguimiento de IA tras link de pago expirado: una sola vez automaticamente. Si el cliente responde, la IA puede continuar el flujo; si no responde, no insiste otra vez en V1.
- 2026-06-09: PRD marcado como final tras resolver preguntas abiertas, reviewer pass alto y decisiones de alcance V1/V2.
