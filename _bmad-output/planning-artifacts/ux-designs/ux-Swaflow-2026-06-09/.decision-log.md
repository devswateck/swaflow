- Dashboard necesita graficas mas profesionales y una lectura comercial mas solida.
- Primero se requiere documentacion para que un agente posterior modifique el frontend con un contrato claro.

- Se cerro el primer paquete UX como contrato final y se agregaron mocks HTML de referencia para Dashboard e Inbox en `.working/`.

## Realidad del producto desde el frontend

- El frontend esta concentrado en un solo archivo grande: `frontend/src/App.tsx`.
- Superficies actuales: Dashboard, Inbox, Productos, Inventario, Ordenes, Citas, Funnels, IA, WhatsApp, Integraciones, Ajustes.
- Los tokens visuales actuales viven principalmente en `frontend/tailwind.config.ts`: `ink`, `line`, `panel`, `brand`, `warn`, `danger`.
- El color de marca actual es verde/teal (`brand: #0f766e`) y el estado activo usa `#e5f3ee`.
- El modo oscuro actual usa parches manuales en `frontend/src/styles.css`.
- Dashboard tiene tarjetas KPI, una lista "Inbox activo", una tira simple de pipeline y paneles laterales, pero no tiene un sistema real de graficas.
- Inbox usa dos columnas: lista de conversaciones y detalle del chat seleccionado. Incluye envio de mensaje, pasar a humano, agendar cita y asignacion de funnel/paso.
- Existen componentes compartidos basicos: `SectionHeader`, `IconButton`, `StatusBadge`, `InfoPanel`, `MessageBubble`, `DataTable`, `Notice`, `ToastStack` e inputs.

## Captura de marca desde la imagen

- Registro visual: tecnologia tactica oscura, alto contraste, acento neon magenta/violeta, marca blanca/lila.
- Fondo dominante: morado casi negro.
- Acento: aro y barra en rango magenta-violeta saturado.
- Marca: geometrica, compacta, mayuscula, fuerte.
- Tagline en la fuente: "TACTICAL INNOVATION".
- Traduccion UX: usar magenta/violeta como acentos de marca, foco y seleccion; no como decoracion masiva. La app debe seguir siendo una herramienta SaaS operativa y legible.

## Stakes

- Calidad de lanzamiento comercial SaaS.
- Usuarios operativos necesitan pantallas densas, legibles y repetibles.
- El producto maneja ventas, conversaciones de WhatsApp, pagos, inventario, citas, IA y aislamiento tenant.

## Form factor

- Primario: web responsiva para desktop/laptop.
- Secundario: fuera de alcance en este ciclo.
- La experiencia valida solo desktop/laptop en esta iteracion.

## Direccion de IA

- Mantener todos los modulos, pero agrupar la navegacion en cuatro zonas:
  - Operacion: Dashboard, Inbox.
  - Comercio: Productos, Inventario, Ordenes, Citas.
  - Automatizacion: IA, Funnels, WhatsApp, Integraciones.
  - Administracion: Ajustes.
- Dashboard e Inbox quedan arriba porque son superficies de trabajo diario.
- La busqueda sigue siendo global; mas adelante puede evolucionar a una navegacion tipo comando.

## Mejoras UX documentadas

- Reemplazar el sistema verde por tokens Swa Tech oscuros/magenta/violeta.
- Madurar modo oscuro, no mantenerlo como parches.
- Agrupar navegacion y clarificar jerarquia por pagina.
- Mejorar Dashboard con deltas KPI, paneles de grafica, conversion de funnel, carga del equipo y salud de integraciones.
- Mejorar Inbox con dos zonas desktop: lista de conversaciones y hilo de mensajes de ancho completo.
- Agregar metadata en chats: tipo de emisor, timestamp, no leidos, responsable, estado IA/humano, funnel y paso.
- Mover acciones de chat a un rail contextual estable, no dejarlas como botones sueltos debajo del hilo.
- Agregar estados vacio, cargando, error, offline, datos desactualizados y permisos.
- Estandarizar componentes compartidos antes de aumentar complejidad visual.
- Usar Recharts para las graficas profesionales del Dashboard.

- [ASSUMPTION] Este turno entrega documentacion lista para un agente frontend, no cambios de codigo.

## Preguntas abiertas

- Confirmar prioridad de implementacion: Inbox primero, Dashboard primero o refresh visual completo primero.

## Escaneo de preocupaciones

- Accesibilidad: acentos neon requieren contraste y foco visibles.
- Marca: evitar convertir la app en una landing morada; gana la legibilidad operativa.
- Densidad: Dashboard e Inbox deben ganar informacion sin volverse ruidosos.
- Dark mode: la implementacion actual es de parches y debe volverse token-driven.
- i18n/copy: los mensajes visibles al usuario deben mantenerse en espanol.
- Motion: minima; los cambios de estado no deben distraer a operadores.
- Offline/realtime: Inbox necesita estados claros de frescura de conexion.
- Pagos: UI no puede insinuar pago confirmado si backend no lo confirma.
- Multi-tenant: UI nunca debe sugerir visibilidad cross-tenant.
